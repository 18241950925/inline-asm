HPU​

硬件架构 4.7更新​
1 整体设计​

本版设计不再假设 HPU 内部存在一组大容量独立 VRF，而是以 统一的本地 SRAM 作为主要数据驻留
层，在 PE 阵列前后仅保留小型流式缓冲，用于吸收访存延迟并支撑流式执行。与此同时，NTT/INTT
与独立 Shuffle 指令在物理实现上复用同一条 128 并行重排/蝶形执行通路，其中 NTT 阶段内部所需的

重排模式由硬件自动给出，而对软件显式可见的 pshuf 仍保留独立扩展能力，以支持自同构及后续
其他重排类操作。该组织方式延续了原文档中“custom0 负责 HPU 内部语义、custom1 负责外部
DMA 配置”的总体边界，但在内部数据组织与执行粒度上进行了面向资源约束的重构。​

1.1 RISC-V核内的HPU接口​

RISC-V 核在译码阶段识别自定义操作码 custom0 和 custom1 。核内对 HPU 的支持仍保持“最小
识别、最小派发”原则：核仅判断指令属于哪一类自定义路径，而不在核内展开 HPU 内部语义。​
对于 custom0 指令，核侧只负责将 HPU 内部命令送入 HPU 的命令队列。由于 custom0 的
opcode 固定，当前设计中更关注的并非完整 32 bit 指令，而是其内部可用编码字段，因此核到 HPU
的主命令接口优先采用 payload 形式，传递 HPU 真正关心的内部编码信息。该命令进入 HPU 指令
队列后，由 HPU 自身完成后续译码、资源检查与执行调度。这样可以继续保持核与 HPU 的解耦，避
免将 HPU 内部执行细节引入核内的常规执行路径。​
对于 custom1 指令，语义边界与 custom0 明确区分。 custom1 主要服务于 主存与 HPU 本地
SRAM 之间的数据搬运配置：CPU / 核侧负责给 DMA 配置主存地址、传输长度、方向及启动信息，而
HPU 侧并不直接感知完整的主存地址与长度参数。与之对应，HPU 只在 SRAM 侧参与 block 级协调，
例如某一数据块是否已被 DMA 占用、何时释放、何时完成等。因此， custom1 实际上形成两条后
端路径：一条通向 DMA 配置面，一条通向 HPU 内部 SRAM 管理器的占用/完成同步逻辑，而不会进
入 custom0 那条计算命令 FIFO。​
在这一接口划分下，RISC-V 核与 HPU 之间仍保持异步并行关系：核在成功派发 custom0 或完成
custom1 的 DMA 配置后即可继续执行后续指令，而 HPU 在本地独立完成计算、重排、采样和数据
搬运过程。这样的分工方式与原版的总体接口边界保持一致，但进一步强化了“核只识别路径，不承
担 HPU 内部状态管理”的设计取向。​

1.2 HPU控制器与执行单元​

HPU 控制器采用“顺序发射骨架 + 局部状态机执行”的组织方式。其前端由命令队列、解码器、轻量
级 scoreboard 和发射控制构成。命令从 cmd_queue 中按序取出后，先由解码器识别指令类别，再
由 scoreboard 检查必要的 RAW/WAW、结构冲突、配置上下文冲突及 SRAM 资源占用情况；满足条件
后，控制器将该指令分派到对应执行路径。首版设计不追求复杂乱序和多发射能力，而以“安全发
射、可验证、可实现”为优先目标。
在数据组织方面，本版 HPU 取消了“大容量独立 VRF + 内部 SRAM”双层数据驻留结构，而改为以 统
一本地 SRAM 作为主要数据层。也就是说，大块多项式数据、中间结果、DMA 搬入的数据以及待处理
对象，均驻留在 HPU 内部的 banked SRAM 中。为保证 PE 阵列能够流式工作，在执行单元前后仅设
置小型 operand/result buffer ，它们的职责不是长期保存完整向量，而是在执行过程中完成
按 chunk 取数、送数和结果暂存。这样既避免了独立大 VRF 带来的容量开销，也更符合当前资源受限
下“数据以本地对象形式驻留、计算按流式窗口推进”的总体思路。
控制器内部还维护一组较小的状态/上下文寄存器，用于保存当前活动模参数、随机数生成状态、以及
独立 Shuffle 所需的模式信息等。与早期方案相比，这里的状态组织更偏向“当前活动上下文”，而不
是维护大量长期并存的复杂状态槽位。为解决 25 bit 内部编码空间有限而又需要指向本地数据对象的

问题，控制器可维护少量本地对象描述信息，用于标识统一本地 SRAM 中待计算或待搬运的数据区
间；这些描述信息是对本地数据对象的命名，而不是另一层独立数据存储。
在执行结构上，本版 HPU 不再将主要计算资源理解为多套彼此独立的大型专用后端，而是将其组织为
一组 可重构的统一 PE 阵列。控制器根据指令类型、当前上下文以及数据通路需求，对该阵列施加不
同配置，使其在不同计算任务下表现出不同的执行模式和物理并行度。
除统一 PE 阵列外，HPU 还包含与其配套的随机/采样执行路径，以及 SRAM 管理与流式访存控制路
径。随机/采样路径面向 pseed 、 psample 等操作，由控制器通过当前活动 seed/state 驱动
PRNG 单元按流式方式产生随机数序列，并将结果写回本地对象对应的数据区域。SRAM 管理与流式访
存控制路径则由 sram_mgr 和相关 load/store 控制逻辑组成，负责统一管理本地 SRAM 的 block 状
态、HPU 与 DMA 的 ownership 关系，以及计算数据从本地 SRAM 到执行缓冲区、再从结果缓冲区写
回本地 SRAM 的流式搬运过程。​
因此，当前 HPU 的执行结构应理解为：以一组可重构统一 PE 阵列为核心计算资源，在不同指令语义
下切换不同执行模式；同时配合小型流式缓冲、随机/采样路径以及统一 SRAM 管理路径，共同构成完
整的 HPU 控制与执行体系。​

1.3 中断驱动的同步机制​

HPU 与 RISC-V 核之间的同步仍通过专用中断线实现，但在本版设计中，同步语义不再主要依附于某一
类具体计算或存储指令，而是通过一条独立的 psync 同步指令 来显式建立软件可见的阶段边界。其
核心思想是：RISC-V 核负责异步派发 HPU 指令流，HPU 在内部顺序发射并独立执行；当软件需要确
认某一批 HPU 内部操作已经全部完成并对后续阶段可见时，可在 custom0 指令流中插入
psync ，由 HPU 在满足同步条件后主动向核侧发出中断通知。这样，计算语义与同步语义得以解
耦，核侧无需轮询 HPU 状态，也不必将中断机制绑定到某一条特定算术或内部访存指令上。​
在这一机制下，RISC-V 核与 HPU 的异步并行关系得以保留。核侧在派发一批 custom0 指令及其后
的 psync 后即可继续执行后续程序；HPU 则在内部独立完成此前命令的执行与收敛。当 psync
对应的同步条件成立后，HPU 再通过中断通知核侧进入软件同步点。这种方式使 psync 成为 HPU
命令流中的显式阶段屏障：它既不会改变前序指令的执行语义，也不会要求核侧在派发后立即阻塞等
待，而是通过“前序收敛—中断通知—软件继续”的方式，建立起适合当前控制架构的同步模型。​
因此，在本版整体设计中，中断机制的主要作用可概括为：不是对单条计算指令完成的被动响应，而
是对 psync 所定义的阶段性完成条件的显式通知。 这一调整使中断驱动同步机制从“依附于特定操
作”转变为“服务于命令流阶段边界”，更符合 HPU 作为独立 IP、采用顺序发射与局部状态机执行的
控制组织方式。

1.4 关键数据通路​

结合上述模块划分，本版 HPU 的关键数据通路主要包括以下几条：​
1. 指令派发通路：RISC-V 核的 Decoder / Dispatcher 将 custom0 命令送入 cmd_queue ，随后
由 HPU 内部完成解码、scoreboard 检查和顺序发射。这一路径保证核侧只承担最小派发责任，而
HPU 内部完整拥有指令语义与资源调度控制权。​

2. DMA 配置与 SRAM 协调通路： custom1 在核侧配置 DMA 后，一方面驱动主存与 HPU 本地
SRAM 之间的数据传输，另一方面通过 block 级同步接口与 sram_mgr 协调对应 SRAM 区块的占
用、释放与完成状态。这样可以保证 HPU 在不掌握主存地址细节的前提下，仍能正确感知本地数
据对象何时可被计算或回写。
3. 本地数据流式执行通路：本地对象数据驻留在统一 SRAM 中。执行时，流式 load/store 控制逻辑按
当前对象描述和操作类型，从本地 SRAM 中连续读取数据到 operand buffer ；随后数据进入
基础向量算术后端或 NTT/Shuffle 复用后端完成计算；结果再经 result buffer 写回统一
SRAM。对基础算术而言，流式粒度按 64 元素组织；对 NTT / Shuffle 而言，流式粒度按 128 并行
重排/蝶形路径组织。这是本版 HPU 的核心计算数据通路。​
4. 状态/上下文控制通路：模参数、随机种子、Shuffle 模式及少量对象描述信息由控制器内部状态寄
存器维护，并在指令执行时提供给各后端。这里的状态并非另一层数据存储，而是用于约束当前运
算模式、数据解释方式和执行后端行为的控制上下文。
5. 中断反馈通路：当 custom0 指令流中的 psync 到达队首后，HPU 控制器停止继续发射其后
的 younger 指令，并等待此前所有已发射的内部操作全部完成且结果对统一 SRAM 可见。待同步条
件满足后，HPU 控制器通过中断线将“阶段完成”信息反馈至核侧中断控制逻辑，随后由软件继续
发起下一阶段计算、配置 DMA，或执行后续处理。这一路径使核侧无需主动轮询 HPU 内部状态，
即可在显式同步点上获得 HPU 执行收敛的通知。​
综上，本版 HPU 的总体设计可以概括为：核侧最小接口、HPU 内部顺序发射控制、统一本地 SRAM
数据层、同一 PE 阵列在不同模式下工作，以及基于中断的异步同步机制。 这一结构既继承了原版中
custom0/custom1 分流、异步执行与真实 Shuffle 的设计思想，又针对当前片上资源约束对内部存储
组织和执行粒度进行了收缩与重构。

2 指令编码和流水线行为​

按照核侧流水线行为划分，HPU 相关指令主要分为两类：一类是通过 custom0 进入 HPU 内部命令
队列的内部指令，包括基础模运算、NTT/INTT stage、Shuffle、随机/采样、模上下文装载以及同步指
令；另一类是通过 custom1 配置外部访存的指令，包括 DLoad 和 DStore ，用于主存与 HPU
本地 SRAM 之间的数据搬运。​
本章仅从 RISC-V 核侧行为 出发描述核内需要支持的译码、派发和字段含义；至于 HPU 内部如何维护
对象槽位、状态寄存器和执行路径，则在后续章节展开。

阶段只需识别该指令属于 HPU 内部命令路径，而不解析其具体语义。随后在 dispatch 阶段将该指令
送入 HPU 的命令队列，由 HPU 内部自行完成译码、资源检查和执行调度。​

1. decode 级识别​

2. dispatch 级派发​

3. retire / complete 级完成​

通用寄存器写回结果。
因此，从核侧视角看， custom0 指令的关键在于“是否成功进入 HPU 队列”，而不是“HPU 是否
已经完成该运算”。

2.2 HPU 外部访存指令（custom1：DLoad / DStore）​

HPU 外部访存指令采用 RISC-V custom1 操作码，用于主存与 HPU 本地 SRAM 之间的数据传输。与
早期仅把 custom1 理解为 DMA 配置不同，本版中 custom1 路径上的两条指令 DLoad 和
DStore 同时承担对象槽位建立/更新的语义：​
DLoad 在把数据搬入 HPU 本地 SRAM 的同时，建立某个活动对象槽位； DStore 在把对象数据导
出到主存的同时，完成对该对象槽位的状态更新或释放。

• rs1 ：存放主存基址的通用寄存器编号 ​

• rd[2:0] ：对象槽位编号 obj_id ​

1. decode 级识别 custom1 ，并将指令送入 HPU​

2.3 基于 psync 的同步行为​

在原始方案中，中断同步主要依附于带中断使能位的 sstore 之类内部访存语义；而在本版中，同
步行为通过专门的 psync 指令来建立，因此第 2 章中的同步描述也应相应更新。​
其流程可概括如下：
1. psync 指令派发​
当 RISC-V 核执行到一条 psync 指令时，该指令作为普通 custom0 命令被前端派发至 HPU 命
令队列。核侧不等待其实际完成，而是继续执行后续指令，除非 HPU 命令队列满而导致 dispatch
停顿。 ​
2. HPU 识别同步屏障​
当 psync 到达 HPU 命令队列队首时，控制器将其识别为同步屏障。此时，HPU 不再发射
psync 之后的 younger custom0 指令，而是等待 psync 之前所有已发射的内部操作收敛完
成。 ​
3. 前序执行收敛
psync 的等待范围包括此前所有已发射的 custom0 内部操作，如基础模运算、NTT / INTT
stage、显式 Shuffle、随机采样及相关结果写回过程。只有当这些操作全部完成，且其结果已经对
HPU 内部统一 SRAM 可见时，控制器才认为同步条件满足。 ​

4. 中断触发
当前序指令全部完成后，HPU 控制器将 psync 标记为完成，并通过专用中断线向 RISC-V 核发送
同步完成中断。该中断所表示的语义是：“此前一批 HPU 内部 custom0 操作已经全部完成，可
进入下一阶段”。 ​
5. 核内中断响应与后续处理
RISC-V 核在中断检测阶段识别到中断有效后，按照常规中断流程保存现场并跳转到中断服务程
序；随后软件可根据当前阶段需要继续发起下一批 HPU 指令、配置 DMA，或执行后续处理逻辑。
这样，中断不再是对某条具体内部访存指令的附属反馈，而是对 psync 所定义的阶段性同步条件
的显式通知。 ​
需要特别指出的是， psync 的同步边界仅覆盖 HPU 内部 custom0 执行域，而不默认等待
custom1 所配置 DMA 传输的完成。DMA 完成仍应通过 DMA 自身的完成反馈和 sram_mgr 的
block 状态更新来体现。这样可以保证内部执行同步与外部数据搬运同步之间的职责分离。​

3. HPU 指令助记符设计​

为支持 HPU 面向多项式模运算、NTT/INTT、随机采样、真实数据重排以及阶段同步等核心流程，本
文定义一组 HPU 指令助记符。它们分为两类：​
• 内部执行类指令：经 custom0 进入 HPU 命令队列，由 HPU 内部执行； ​
• 外部访存类指令：经 custom1 发起主存与 HPU 本地 SRAM 之间的数据搬运，并建立或更新活动
对象槽位。 ​
与早期方案相比，本版不再以“大容量向量寄存器文件 + 内部 SRAM”的双层数据组织为前提，而是
以 统一 SRAM + 少量活动对象槽位 为核心数据组织方式。​

3.1 设计原则​

本版指令集遵循以下原则：
1. 面向本地数据对象组织
在当前资源约束下，HPU 内部不再适合继续采用“大容量独立 VRF + 内部 SRAM”的双层数据层次，
而改为以 统一本地 SRAM 作为主要数据驻留层。计算、重排、采样等操作均围绕本地数据对象展开，
PE 阵列前后仅保留小型流式缓冲，用于支持数据按片段连续送入和结果回写。因此，本版指令中的源
和目的不再优先理解为“大容量向量寄存器”，而应理解为对本地 SRAM 中当前活动对象或对象片段
的引用。
2. 对象描述符表示活动对象/片段视图​
本版中的 pdst 、 psrc1 、 psrc2 、 psrc 等字段并不表示传统意义上的向量寄存器号，而表
示 活动对象/片段视图描述符。​
这些描述符由 HPU 控制状态维护，可指向：​
• 一个完整多项式对象； ​
• 一个对象的局部片段； ​

一个对象在当前执行模式下的逻辑视图。 ​
其中， pntt / pintt 等需要全局数据依赖的指令，应作用于完整多项式对象；而 padd /
psub / pmul / pmac / pshuf / psample 等指令，则可以作用于片段对象，以适应资源受
限条件下的流式访存和局部计算。
3. Shuffle 为真实重排操作​
Shuffle 仍然是本版指令集中的核心能力之一。其原因与原文档保持一致：NTT、自同构以及其他提取/
重排类操作均要求在 HPU 内部对数据顺序进行真实调整，而不是仅在地址解释层做逻辑映射。因此，
本版仍保留独立的 Shuffle 指令能力。不同之处在于：对于 NTT/INTT 执行过程中所需的部分重排，硬
件可以在内部复用同一条 Shuffle 数据通路而不要求软件逐次显式配置；但对于自同构和后续可扩展的
重排类操作， pshuf 仍然作为软件可见的独立指令保留。​
4. NTT / INTT 采用 stage 级指令粒度​
本版中， pntt 和 pintt 表示单个 stage 的蝶形执行。这样做的原因在于：一次完整变换需要经历
多个 stage，每个 stage 内又包含 butterfly 计算、twiddle 推进以及必要重排；若将整个过程压缩为单
条指令，则会使单条指令语义过于庞大，不利于控制器和软件之间建立清晰边界。因此，本版采
用“软件显式推进 stage，硬件在 stage 内部自动完成细节”的语义划分：软件负责发起当前 stage，
硬件负责在该 stage 内推进 twiddle、完成 butterfly 计算并调用必要的内部重排。​
5. 同步语义从具体操作中解耦
原方案中的中断同步主要依附于某些具体指令；而本版中，同步能力通过独立的 psync 指令显式提
供。这样，计算、重排、采样等功能指令与阶段同步边界被分离：软件可在一批 HPU 指令之后插入
psync ，要求 HPU 在前序操作全部完成后向核侧发出同步完成中断。这使同步语义不再绑定于某一
类具体运算，而成为 HPU 命令流中的显式屏障能力。​
•

常量、小控制状态与大数据对象分离处理​

本版将“大数据对象”和“小控制状态”分开处理：
• 大数据对象通过 DLoad / DStore 建立为对象槽位； ​
• 小常量优先通过指令控制字段直接编码； ​
• 模上下文通过 pmodld 从对象槽位装载； ​
• pseed 直接采用立即数种子形式； ​
• Shuffle 配置通过 pshcfg 从对象槽位装载。 ​

4.6 本版编码方案的设计理由​

综上，本版第 4 章相较上一版有三项关键调整：​
1. 扁平化主操作码
不再使用稀疏的 CLASS + OP 两级结构，而改用 4 bit 主操作码。​
这样节省出的位宽优先用于模式、片段控制和同步标记，而不是继续扩大操作码树。​
2. 对象描述符仍保持 3 bit​
对象描述符不再被解释为“很多完整多项式对象编号”，而被解释为当前活动对象/片段视图槽位。​
因此，保留 3 bit 是合理的；继续扩宽并不能直接带来现实收益。​
3. 明确区分“完整对象”与“片段对象”
• pntt / pintt 面向完整多项式对象 ​
• padd / psub / pmul / pmac / pshuf / psample 可面向片段对象 ​
4. pseed 保留在配置类中，但 inst[27:7] 整段解释为 IMM21 ； ​
5. pmodld 改为从对象槽位中装载模上下文，且该模上下文中可包含 twiddle 静态参数； ​
6. pshcfg 改为从对象槽位中装载 Shuffle 配置； ​
7. dload / dstore 被纳入统一编码体系，其中 dload 可区分普通片段对象、完整多项式对
象、模上下文对象和 Shuffle 配置对象。​
这样编码语义更贴近当前硬件存储现实，也更适合统一 SRAM + 流式执行的架构。​
